# 판례 AI 플랫폼 - 사용자 플로우 v2

> 이 문서는 새 기획 기준으로 작성한 사용자 플로우 및 페이지 설계서입니다.
> 기존 팀 프로젝트(판례 요약 시스템)의 연장선이며, 확장된 서비스 구조를 반영합니다.

---

## 1. 서비스 개요

AI 기반 법률 판례 서비스. 두 가지 핵심 기능을 제공한다.

- **챗봇**: 사용자가 사례를 입력하면 RAG(ChromaDB)로 관련 판례를 검색하여 해석과 원본 링크를 제공
- **워크스페이스**: 프리미엄 사용자가 그룹 단위로 문서를 업로드하고 AI 요약을 열람

판례 목록을 직접 브라우징하거나 검색하는 기능은 제공하지 않는다.
판례 원본은 외부 URL 링크로만 제공하며, 챗봇 응답 안에 인라인으로 포함된다.

## 2. 사용자 유형 및 접근 권한

| 유형                 | 조건                     | 접근 가능 페이지           |
| -------------------- | ------------------------ | -------------------------- |
| 비로그인             | -                        | 랜딩(챗봇)                 |
| 일반 (`GENERAL`)     | 로그인                   | 랜딩(챗봇) + 마이페이지    |
| 프리미엄 (`PREMIUM`) | 로그인 + 플랜 업그레이드 | 일반 + 워크스페이스 + 문서 |
| 관리자 (`ADMIN`)     | role=ADMIN               | 전체 + 어드민 페이지       |

## 4. 플로우 상세

### 4-1. 랜딩 페이지 (`/`)

```
레이아웃
├── 헤더 (로고 + 로그인/마이페이지 버튼)
├── 사이드바 (로그인 시에만 노출)
│   └── 챗봇 히스토리 세션 목록
│       ├── 세션 클릭 → 해당 대화 불러오기
│       └── 세션 삭제 버튼
└── 챗봇 메인 영역
    ├── 입력창
    ├── 전송 버튼
    └── 대화 내용 표시 (판례 원본 링크 인라인 포함)
```

**비로그인 상태**

- 챗봇 사용 가능
- 히스토리 사이드바 미노출
- 대화 내용 세션 저장 안 됨

**로그인 상태**

- 챗봇 사용 가능
- 히스토리 사이드바 노출
- 대화 내용 DB 저장 (`chat_sessions` / `chat_messages`)
- 세션 직접 삭제 가능

**챗봇 응답 흐름**

```
사용자 입력
└── POST /api/chat/sessions/{session_id}/messages
    └── 백엔드 내부
        ├── ChromaDB RAG 검색 (판례 벡터)
        ├── EXAONE 모델 추론
        └── 응답 반환 (해석 + 판례 원본 URL 인라인)
```

### 4-2. 인증 플로우

**로그인**

```
로그인 모달 (헤더 버튼 또는 보호 페이지 접근 시 팝업)
├── 이메일 + 비밀번호 입력
│   ├── 실패 (5회 초과) → 5분 잠금 (Redis rate limit)
│   └── 성공 → access_token + refresh_token 쿠키 설정
└── /api/auth/me로 사용자 정보 복원
```

**회원가입**

```
회원가입 모달
├── 이메일 인증번호 발송 → POST /api/email/send-verification-code
├── 인증번호 확인 → POST /api/email/verify-code
├── 이름 + 비밀번호 입력
└── POST /api/auth/signup
```

**토큰 관리**

- `access_token`: Cookie (httpOnly: false)
- `refresh_token`: Cookie (httpOnly: true)
- Axios 인터셉터가 만료 시 `/api/auth/refresh` 자동 재시도

### 4-3. 워크스페이스 플로우 (`/workspace`)

프리미엄 사용자 전용. 그룹 = 워크스페이스.

| 역할   | 누구                        |
| ------ | --------------------------- |
| OWNER  | 회사 대표 / 법무팀 책임자   |
| ADMIN  | 법무팀 관리자               |
| EDITOR | 법무팀 실무자 / 외부 변호사 |
| VIEWER | 일반 직원                   |

그룹 없음 → "워크스페이스를 만들어보세요" 빈 상태 화면 + 생성 버튼

"유저 1명당 OWNER 가능한 그룹은 1개"

누구든 여러 그룹에 멤버로 속할 수 있다
→ 목록 페이지는 있어야 한다
예시) 프리랜서 법률 전문가같은 경우 임시로 멤버 초대 받을 수 있는데 동시에 여러 회사에서 일할 수 있기때문

그룹 카드에 오너 누군지도 표시

단, 기존 그룹을 삭제(30일 유예 완료)하면 다시 생성 가능
복구할 때 기존 ACTIVE 그룹이 있으면 복구 불가

그룹 삭제 요청 시
그룹 전체 멤버 + 문서 업로더에게 알림 발송
그룹 삭제 요청 후 → 워크스페이스 화면에 "삭제 예정 D-23, 취소하기" 배너

```
[워크스페이스 삭제 예정]
워크스페이스 "abc"이 삭제 예정입니다.
30일 후 모든 문서가 영구 삭제됩니다.

필요한 파일을 백업해주세요.
삭제 예정일: 2024-04-05
```

삭제 후 그룹 페이지에 접근은 가능함
단,
`⚠ 워크스페이스 삭제 예정` 상태 메세지 띄우기```
문서 조회 ✅
문서 다운로드✅
업로드 ❌
멤버 초대 ❌

초대 링크 수락 방식 → 대기 중 표시

항목 결정
과금 오너(대표)만 프리미엄, 초대 멤버는 무료여도 가능
그룹 회사 단위 1그룹 = 워크스페이스
OWNER 제한 1인 1그룹

권한 업로드 버튼 멤버 초대 삭제 버튼 멤버 관리 탭
OWNER ✅ ✅ ✅ (모든 문서) ✅
ADMIN ✅ ✅ ✅ (오너제외문서) ✅
EDITOR ✅ ❌ ✅(본인것만) ❌
VIEWER ❌ ❌ ❌ ❌

삭제 문서는 삭제 탭으로

문서목록에서도 원본 다운로드, 삭제 버튼 추가(호버시)
[전체 문서 다운로드] 버튼 누르면 전체 파일 zip으로 다운로드, 유출 방지를 위해 EDITOR, VIEWER는 전체 다운은 제한, 개별 다운은 가능
선택 다운로드

보안을 위해 다운로드 로그 추가할지(시간 남으면)

문서 보안 등급 추가할건지?
| 보안 등급 | 의미 |
| ------------ | ----- |
| CONFIDENTIAL | 내부 문서 |
| STRICT | 매우 민감 |

문서 보안 등급 적용한 전체 다운로드(ZIP) 정책
| 권한 | CONFIDENTIAL | STRICT |
| ------ | ------------ | ------ |
| OWNER | 다운로드 | 다운로드 |
| ADMIN | 다운로드 | 다운로드 |
| EDITOR | 다운로드 | ❌ |
| VIEWER | ❌ | ❌ |

**워크스페이스 목록**

```
/workspace
├── 내가 속한 그룹 목록 조회 → GET /api/groups
├── 그룹 생성 버튼 → POST /api/groups
└── 그룹 카드 클릭 → /workspace/:group_id
```

POST /api/groups # 그룹 생성
GET /api/groups/{group_id} # 그룹 상세(메인 화면)
DELETE /api/groups/{group_id} # 그룹 삭제 요청 (30일 유예)
POST /api/groups/{group_id}/cancel-delete # 그룹 삭제 취소

GET /api/groups/{group_id}/members # 멤버 목록
POST /api/groups/{group_id}/members # 멤버 초대(OWNER, ADMIN)
PATCH /api/groups/{group_id}/members/{user_id} # 권한 변경
DELETE /api/groups/{group_id}/members/{user_id} # 멤버 추방

GET /api/groups/{group_id}/documents?category=&keyword= # 그룹 문서 목록 (카테고리 필터)
DELETE /api/groups/{group_id}/documents/{doc_id} # 문서 삭제 요청 (7일 유예, 알림)
POST /api/groups/{group_id}/documents/{doc_id}/restore # 문서 삭제 취소 (7일 내 복구)

```


**그룹 상세 (문서 목록)**
```

/workspace/:group_id
├── 그룹 정보 조회 → GET /api/groups/{group_id}
├── 문서 목록 조회 → GET /api/groups/{group_id}/documents?category=&keyword=
│ └── 카테고리 필터: 전체 / 민사 / 형사 / 노동 / 행정 / 가사
├── 문서 카드 클릭 → /workspace/:group_id/documents/:doc_id
├── 업로드 버튼 → /workspace/:group_id/upload
└── 멤버 관리 버튼 (OWNER/ADMIN만 노출)
└── GET /api/groups/{group_id}/members
├── 초대 → POST /api/groups/{group_id}/members
├── 권한 변경 → PATCH /api/groups/{group_id}/members/{user_id}
└── 추방 → DELETE /api/groups/{group_id}/members/{user_id}

```

**권한 등급**
```

OWNER > ADMIN > EDITOR > VIEWER

- OWNER: 모든 권한, 워크스페이스당 1명, 본인 등급 변경 불가
- ADMIN: EDITOR/VIEWER 권한만 변경 가능
- EDITOR/VIEWER: 권한 변경 불가

```

**업로드 파이프라인 구분**

- 그룹 문서 업로드
  - 대상: 워크스페이스 문서
  - 목적: 영속 저장 + AI 요약
  - 저장: `documents`, `summaries`, 원본 파일 저장소
  - 상태: `Document.processing_status`, Redis 업로드 세션
- 판례 업로드
  - 대상: RAG용 판례 데이터
  - 목적: 판례 메타 저장 + 인덱싱
  - 저장: `precedents`, 벡터/BM25 인덱스
  - 상태: `Precedent.processing_status`
- 채팅 임시 업로드
  - 대상: 채팅 중 참고용 파일
  - 목적: 일회성 컨텍스트 확보
  - 저장: 임시 공간
  - 상태: 세션 범위 임시 상태만 관리

**문서 업로드**
```

/workspace/:group_id/upload
├── PDF 파일 선택 (드래그 앤 드롭 또는 탐색기)
│ ├── PDF만 허용
│ └── 최대 5개
├── 업로드 세션 생성 → POST /api/documents/upload-session
├── 파일별 순차 업로드 → POST /api/documents/upload (file + group_id)
├── 백엔드(group_document_upload)
│ ├── 그룹 권한 확인 (OWNER / ADMIN / EDITOR)
│ ├── 원본 파일 저장
│ ├── documents row를 PENDING으로 생성
│ ├── 업로드 세션 상태를 processing으로 갱신
│ ├── Celery 요약 작업 enqueue
│ └── 업로드 상태 WebSocket publish
├── worker
│ ├── 문서 상태 PROCESSING 반영
│ ├── OCR fallback
│ ├── LLM 요약
│ ├── summary 저장
│ └── 문서 상태 DONE / FAILED 반영
└── 프론트
  ├── WebSocket 수신
  ├── 처리 현황 업데이트
  └── 완료 시 아코디언으로 요약 확인

```

- 핵심 원칙:
  - 업로드 요청은 접수와 저장까지만 빠르게 끝낸다.
  - OCR/요약 같은 무거운 처리는 Celery worker가 비동기로 수행한다.
  - 사용자 진행 상태는 Redis 업로드 세션 + WebSocket으로 복구/동기화한다.

**문서 상세**
```

/workspace/:group_id/documents/:doc_id
├── GET /api/documents/{doc_id}
├── 사건번호, 법원, 판결일, AI 요약 표시
├── PDF 다운로드 → GET /api/summaries/{summary_id}/download
└── 문서 삭제 버튼 (권한 있는 경우만 노출)
└── DELETE /api/groups/{group_id}/documents/{doc_id}
├── 높은 권한자가 낮은 권한자 문서 삭제 시 7일 유예
├── 문서 소유자에게 알림 발송
└── 7일 내 복구 가능 → POST /api/groups/{group_id}/documents/{doc_id}/restore

```

**그룹 삭제 플로우**
```

DELETE /api/groups/{group_id} (OWNER만 가능)
├── 그룹 멤버 전체에게 알림 발송
├── 30일 유예 (soft delete)
├── 30일 내 취소 가능 → POST /api/groups/{group_id}/cancel-delete
└── 30일 후 그룹 + 문서 영구 삭제

```

### 4-5. 어드민 페이지 (`/admin`)

ADMIN 역할만 접근 가능.

어드민은 플랫폼 운영자 관점의 화면이다.
- 고객 워크스페이스 내부 문서/권한/승인 흐름에는 직접 개입하지 않는다.
- 서비스 전체 운영 지표, 회원/구독 상태, RAG용 판례 데이터베이스를 관리한다.

```

/admin
├── 대시보드 탭
│ └── 통계 카드 → GET /api/admin/stats
│ (총 유저 수, PREMIUM 전환율, 활성 워크스페이스 수, AI 처리 성공률 등)
├── 사용량 탭
│ ├── 서비스 사용량 → GET /api/admin/usage
│ │ (워크스페이스 업로드 용량, 최근 7일 업로드 추이, 문서 처리량)
│ └── RAG 사용량 → GET /api/admin/usage
│ (등록 판례 수, 인덱싱 상태, 벡터 저장량)
├── 사용자 관리 탭
│ ├── 전체 유저 목록 → GET /api/admin/users
│ ├── 활성화/비활성화 → PATCH /api/admin/users/{user_id}
│ └── 유저 삭제 → DELETE /api/admin/users/{user_id}
├── RAG DB 관리 탭
│ ├── 전체 판례 목록 → GET /api/admin/precedents
│ ├── 판례 URL 등록 → POST /api/admin/precedents
│ ├── 실패 항목 재처리 → POST /api/admin/precedents/{precedent_id}/retry
│ └── 인덱스 재생성 → POST /api/admin/precedents/reindex

```

## 6. 인프라 구성

```

yourdomain.com/ → Nginx → Vite 빌드 결과물 (정적 파일)
yourdomain.com/api/ → Nginx → FastAPI :8000
yourdomain.com/ws/ → Nginx → FastAPI WebSocket (챗봇 스트리밍)

````

**Docker 컨테이너**
- `backend`: FastAPI
- `frontend`: Nginx (정적 파일 서빙)
- `db`: MariaDB
- `redis`: Redis
- `ollama`: EXAONE 모델
- `chromadb`: 판례 벡터 DB

**배포 순서**
```bash
cd frontend && npm run build   # 프론트 빌드
docker-compose up -d           # 전체 컨테이너 실행
````

## 7. DB 테이블 요약

| 테이블          | 설명                                     |
| --------------- | ---------------------------------------- |
| `users`         | 사용자 (role: GENERAL/ADMIN)             |
| `subscriptions` | 구독 플랜 (FREE/PREMIUM, 형식만)         |
| `groups`        | 워크스페이스 (soft delete, 30일 유예)    |
| `user_groups`   | 유저-그룹 다대다 (권한 등급 포함)        |
| `documents`     | 업로드 문서 (group_id NOT NULL)          |
| `summaries`     | AI 요약 결과                             |
| `precedents`    | RAG용 판례 메타 + 처리 상태 관리 테이블  |
| `chat_sessions` | 챗봇 세션 (로그인 유저만)                |
| `chat_messages` | 챗봇 메시지 (role: user/assistant)       |
| `notifications` | 서비스 내 알림 (그룹 삭제, 문서 삭제 등) |

`precedents` 운영 원칙:

- 어드민은 지원된 도메인의 판례 URL만 등록한다.
- 서버가 URL을 기준으로 원문을 수집/벡터화하여 ChromaDB에 저장한다.
- RDB의 `precedents`는 운영용 메타 및 처리 상태를 관리한다.
- 실패 항목은 row를 유지한 채 재처리할 수 있다.
