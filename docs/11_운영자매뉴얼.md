# 운영자 매뉴얼

## 1. 문서 개요

- 시스템명: RAG기반 법률 상담 및 문서 요약 시스템
- 문서 목적:
  - 운영자가 서비스를 배포, 실행, 점검, 설정할 수 있도록 안내한다.
  - LLM, RAG, 환경설정, 운영 점검 절차를 정리한다.
- 대상:
  - 운영자
  - 백엔드 개발자
  - AI/RAG 담당자
  - PM 및 기술 지원 인력

## 2. 시스템 구성 개요

본 시스템은 다음 구성요소로 운영된다.

- 프론트엔드: React, Vite
- 백엔드: FastAPI
- 데이터베이스: MariaDB
- 캐시 및 큐 브로커: Redis
- 비동기 작업 처리: Celery Worker
- 벡터 및 검색 저장소: Qdrant, BM25 저장소
- LLM 연동: Ollama 또는 외부 OpenAI-compatible API
- 배포 인프라: Docker Compose, Nginx, Cloudflare Tunnel

## 4. 애플리케이션 구동 절차

### 4.1 개발 환경 구동

1. 프로젝트 루트로 이동한다.
2. 환경변수를 설정한다.
3. 아래 명령으로 개발 환경을 실행한다.

```bash
docker compose up -d
```

4. 필요 시 프론트엔드와 백엔드를 개별 실행한다.

프론트엔드:

```bash
cd frontend
npm install
npm run dev
```

백엔드:

```bash
cd backend
pip install -r requirements.txt
uvicorn main:app --reload
```

### 4.2 서버 환경 구동

운영 환경은 `docker-compose.server.yml`을 overlay 방식으로 사용한다.

```bash
docker compose -f docker-compose.yml -f docker-compose.server.yml up -d --remove-orphans
```

### 4.3 정상 구동 확인

운영자는 다음 항목을 확인해야 한다.

- 프론트엔드 접속 가능 여부
- 백엔드 API 응답 여부
- `/api/health` 정상 응답 여부
- DB, Redis, Qdrant 연결 여부
- Celery Worker 정상 실행 여부

## 5. LLM 구동 절차

### 5.1 개요

본 시스템의 답변 생성과 요약 생성은 LLM 설정에 따라 동작한다. 운영 환경에서는 Ollama 또는 외부 OpenAI-compatible API를 사용할 수 있다.

### 5.2 Ollama 기반 구동

1. Ollama 서버를 실행한다.
2. 사용할 모델을 사전에 pull 한다.
3. 백엔드 환경변수에 LLM 연결 정보를 설정한다.

예시:

```bash
ollama serve
ollama pull qwen2.5:3b
```

환경변수 예시:

```env
OLLAMA_HOST=http://host.docker.internal:11434
OLLAMA_MODEL=qwen2.5:3b
OLLAMA_NUM_CTX=8192
OLLAMA_NUM_PREDICT=3072
OLLAMA_TIMEOUT_SECONDS=600
```

### 5.3 외부 엔드포인트 기반 구동

외부 LLM 서버 또는 OpenAI-compatible API를 사용할 경우 운영 환경변수에서 엔드포인트를 지정한다.

운영 시 확인 항목:

- 엔드포인트 접근 가능 여부
- 모델명 일치 여부
- 응답 시간 및 timeout 설정
- max context, max output 설정

### 5.4 운영 체크포인트

- 요약 생성이 정상 동작하는지 확인한다.
- 챗봇 응답이 정상 생성되는지 확인한다.
- timeout 발생 여부를 로그에서 확인한다.
- 모델 변경 시 응답 품질과 처리 속도를 함께 점검한다.

## 6. RAG 설정 부분 기술

### 6.1 RAG 구성 개요

본 시스템의 RAG는 크게 두 영역으로 구성된다.

- 플랫폼 지식 검색
  - 법률 및 판례 데이터 검색
- 워크스페이스 문서 검색
  - 사용자가 업로드한 문서 검색

검색은 다음 구성요소를 사용한다.

- 임베딩 생성
- 벡터 저장소(Qdrant)
- 키워드 검색(BM25)
- 결과 병합 및 컨텍스트 조립

### 6.2 워크스페이스 문서 인덱싱 절차

1. PDF 업로드
2. 문서 텍스트 추출
3. ODL hybrid OCR 기반 추출 수행
4. 문서 정규화
5. chunk 생성
6. 임베딩 생성
7. 벡터 저장소 저장
8. BM25 저장소 저장

### 6.3 플랫폼 지식 인덱싱 절차

1. 원문 또는 메타데이터 수집
2. 전처리 및 구조화
3. chunk 생성
4. 임베딩 생성
5. 벡터 및 검색 저장소 적재

### 6.4 주요 설정 항목

운영자는 다음 항목을 조정할 수 있다.

- retrieval top-k
- source별 context 길이 제한
- dedupe 기준
- dense / keyword / hybrid 검색 모드
- 임베딩 모델 설정
- 벡터 저장소 연결 정보

### 6.5 운영 체크포인트

- 검색 결과가 비정상적으로 적거나 많은지 확인한다.
- 특정 문서가 검색되지 않으면 인덱싱 상태를 확인한다.
- chunk 수와 임베딩 생성 실패 여부를 점검한다.
- 플랫폼 지식과 워크스페이스 문서가 의도한 범위로 분리 검색되는지 확인한다.

## 7. 기타 설정 부분 기술

### 7.1 인증 설정

주요 환경변수:

```env
JWT_SECRET_KEY=CHANGE_ME
JWT_ALGORITHM=HS256
JWT_ACCESS_TOKEN_EXPIRE_MINUTES=30
JWT_REFRESH_TOKEN_EXPIRE_DAYS=7
```

운영 점검 항목:

- 액세스 토큰 만료 후 refresh 동작 여부
- 인증 실패 시 예외 응답 형식 정상 여부
- 보안 정책에 맞는 쿠키 설정 여부

### 7.2 CORS 설정

주요 환경변수:

```env
CORS_ALLOW_ORIGINS=https://example.com
```

운영 점검 항목:

- 허용 origin이 정확히 설정되었는지 확인한다.
- 운영 도메인과 프론트엔드 URL이 일치하는지 확인한다.

### 7.3 데이터베이스 설정

주요 환경변수:

```env
DATABASE_URL=mysql+pymysql://user:password@db:3306/database_name
```

운영 점검 항목:

- DB 연결 여부
- migration 적용 여부 (`backend_migrate` 성공 또는 `alembic upgrade head` 수동 실행)
- 읽기/쓰기 권한 여부

### 7.4 Redis 설정

주요 환경변수:

```env
REDIS_HOST=redis
REDIS_PORT=6379
```

운영 점검 항목:

- 인증/세션/캐시 동작 여부
- Celery broker 연결 여부

### 7.5 업로드 및 파일 저장 설정

운영자는 다음 항목을 확인해야 한다.

- 업로드 디렉터리 생성 여부
- 파일 저장 권한 여부
- 용량 제한 정책 적용 여부
- 백업 및 삭제 정책

### 7.6 알림 및 운영 로그

운영자는 다음 항목을 점검해야 한다.

- 알림 생성 실패 여부
- 비동기 작업 실패 로그
- 외부 API 호출 실패 로그
- 업로드/요약/검색 오류 로그
