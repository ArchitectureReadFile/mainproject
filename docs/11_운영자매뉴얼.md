# 운영자 매뉴얼

## 1. 목적

이 문서는 운영자가 현재 서비스 구조를 기준으로 다음을 수행할 수 있게 정리한 문서입니다.

- 서비스 기동
- 배포 및 재기동
- migration 확인
- worker/queue 상태 점검
- 문서 처리, 채팅, 플랫폼 동기화 이상 징후 확인

## 2. 현재 운영 구조

### 2.1 핵심 구성

- Frontend: React/Vite 결과물 서빙
- Backend: FastAPI API
- Async: Celery + Redis
- Database: MariaDB
- Search: Qdrant + BM25
- AI: ODL Hybrid, Ollama 또는 외부 OpenAI-compatible API

### 2.2 runtime 규칙

- 앱 시작 시 DB 스키마를 자동 생성하지 않음
- `backend_migrate`가 먼저 실행됨
- migration 성공 후 backend/worker가 올라감
- 세션 첨부 문서는 `reference-upload` 비동기 경로로만 처리됨

## 3. 로컬 개발 운영

### 3.1 기본 실행

```bash
docker compose up -d --build
```

### 3.2 worker 포함 실행

```bash
docker compose --profile dev up -d --build
```

또는:

```bash
docker compose --profile dev up -d celery_worker
```

### 3.3 정상 상태

- `backend_migrate_container` → `Exited (0)`
- `backend_container` → `Up`
- `celery_beat_container` → `Up`
- `celery_worker_container` → `Up` (worker를 띄운 경우)

## 4. 서버 운영

### 4.1 실행 명령

```bash
docker compose -f docker-compose.yml -f docker-compose.server.yml up -d --remove-orphans
```

### 4.2 서버 전용 env

운영 시크릿은 repo 밖 파일에서 관리합니다.

```text
D:/jiyun-env/backend.env
```

핵심 값:

- `DATABASE_URL`
- `MARIADB_ROOT_PASSWORD`
- `MARIADB_DATABASE`
- `MARIADB_USER`
- `MARIADB_PASSWORD`
- `JWT_SECRET_KEY`
- `REDIS_HOST`
- `QDRANT_HOST`
- `ODL_HYBRID_URL`

### 4.3 서버 worker 구조

- `chat_worker`
- `chat_reference_worker`
- `document_worker`
- `export_worker`
- `platform_worker`
- `platform_sync_worker`
- `maintenance_worker`
- `celery_beat`

## 5. migration 운영

### 5.1 정상 동작

`backend_migrate`는 계속 떠 있는 서비스가 아닙니다.

정상 상태:
- 실행
- `alembic upgrade head`
- 성공 후 `Exited (0)`

### 5.2 수동 확인

```bash
docker logs --tail 200 backend_migrate_container
```

정상 예시:
- `Detected pre-Alembic schema. Stamping current schema to head.`
- 또는 일반 `upgrade head` 성공 로그

### 5.3 수동 적용

필요하면 backend 환경에서 직접 실행합니다.

```bash
cd backend
alembic upgrade head
```

## 6. 운영 점검 포인트

### 6.1 backend

확인 항목:
- API 응답 여부
- startup 완료 여부
- DB, Redis, Qdrant 연결 오류 여부

로그:

```bash
docker logs --tail 200 backend_container
```

### 6.2 Celery

확인 항목:
- worker 상태
- queue 바인딩
- task 소비 여부

로그:

```bash
docker logs --tail 200 celery_worker_container
docker logs --tail 200 celery_beat_container
```

### 6.3 문서 처리

확인 항목:
- `document_queue` 소비 여부
- `process_next_pending_document` 반복 실패 여부
- ODL Hybrid 연결 여부

### 6.4 채팅

확인 항목:
- `chat_queue`, `chat_reference_queue` 소비 여부
- 세션 첨부가 `PROCESSING`에서 멈추지 않는지
- 같은 세션 동시 전송 차단이 동작하는지

### 6.5 플랫폼 동기화

확인 항목:
- `platform_sync_queue`
- `platform_queue`
- sync run 실패 여부

## 7. 자주 보는 이상 징후

### 7.1 worker가 안 뜰 때

로컬에서는 `dev` profile 누락 가능성이 가장 큽니다.

```bash
docker compose --profile dev up -d celery_worker
```

### 7.2 backend_migrate가 실패할 때

우선 확인:
- DB health
- env 파일
- `backend_migrate_container` 로그

기존 스키마가 이미 있는 DB라면 stamp 로그가 보일 수 있습니다. 이건 정상입니다.

### 7.3 채팅 첨부가 반영되지 않을 때

확인 순서:

1. `chat_reference_worker` 또는 로컬 `celery_worker` 상태
2. `chat_session_references.status`
3. backend 로그
4. ODL Hybrid 로그

### 7.4 검색/답변이 비정상일 때

확인 순서:

1. Qdrant 상태
2. Redis 상태
3. worker 상태
4. backend retrieval/LLM 로그

## 8. 운영 원칙

- migration 없이 backend/worker를 직접 띄우지 않습니다.
- 운영 비밀번호와 시크릿은 compose 파일에 하드코딩하지 않습니다.
- 세션 첨부/legacy 계약은 우회하지 않습니다.
- source of truth가 정리된 구조를 문서와 운영 절차에서도 그대로 유지합니다.
